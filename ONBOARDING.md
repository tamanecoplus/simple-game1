# 新規参加者向けオンボーディングガイド

## 1. このリポジトリの現状

現在の `simple-game1` リポジトリは、Git 管理のみ初期化された**最小構成のスケルトン**です。

- ルート直下には `.gitkeep` のみがあります。
- アプリ本体コード（`src/` など）、ビルド設定、テストコード、依存関係定義ファイルはまだ配置されていません。

## 2. コードベースの一般的な構造（これから作るべき標準形）

現時点で実ファイルはありませんが、ゲーム開発で保守しやすい構成として次のような形を推奨します。

- `src/` : ゲーム本体ロジック
  - `core/` : ゲームループ、状態管理、ルール
  - `entities/` : プレイヤー・敵・オブジェクト
  - `systems/` : 入力、描画、物理、衝突判定
- `assets/` : 画像、音声、フォント
- `tests/` : 単体・結合テスト
- `docs/` : 仕様、設計、運用手順
- `README.md` : セットアップ、実行方法、開発フロー

## 3. 参加者がまず知っておくべき重要事項

1. **仕様ドキュメントが未整備**
   - まず最小仕様（目的、勝敗条件、入力方法）を決めて固定する必要があります。
2. **技術スタック未確定**
   - 言語・フレームワーク（例: TypeScript + Phaser、Unity、Godot など）を確定しないとディレクトリ構成も最適化できません。
3. **開発ルール未定義**
   - ブランチ運用、コミット規約、コード規約、レビュー基準を早期に決めると後の手戻りを防げます。
4. **テストとCIが未設定**
   - 初期段階でテスト実行と静的解析を自動化しておくと、開発速度と品質が安定します。

## 4. 次に学習・着手するとよい順序

1. **ゲームの最小仕様を定義**（1ページで十分）
2. **技術スタックを選定**（将来の配布先を基準に）
3. **プロジェクト骨格を作成**（`src/`, `tests/`, `README.md`）
4. **最小ゲームループを実装**（入力→更新→描画）
5. **テスト/CIを導入**（最低限の自動チェック）
6. **機能追加の優先順位付け**（操作感→ゲーム性→演出）

## 5. 新規参加者向けチェックリスト
